# Architecture

## Service Provider Pattern

The theme uses a Service Provider pattern for initializing classes:

```
functions.php → App.php → ServiceProviders initialized
```

- `functions.php` — loads Composer autoloader and instantiates `App`
- `App.php` — instantiates each provider in `App::$service_providers`
- Service providers are organized by domain under `includes/classes/`:
  - `Rest/RestServiceProvider.php` — boots REST-related services
  - `Theme/ThemeServiceProvider.php` — boots theme/admin services
  - ...
- Services register their own WordPress hooks in their constructors

Current class structure:

```
includes/classes/
├── App.php
├── Rest/
│   ├── RestServiceProvider.php
│   ├── SocialNetworksRestEndpoint.php
│   └── Add new feature
├── Theme/
│   ├── ThemeServiceProvider.php
│   ├── ThemeOptions.php
│   └── Add new feature
└── NewDomain/                  # Add new service provider folder/group here.
    ├── NewDomainServiceProvider.php
    └── NewFeature.php
```

## PSR-4 Autoloading

Classes follow PSR-4 autoloading via Composer:

```
RVStarterTheme\ → includes/classes/
```

Update `composer.json` and run `composer dump-autoload` when adding new classes.

## Namespaced Functions

Non-class code is organized as namespaced functions in `includes/`:

| File | Purpose |
|------|---------|
| `core.php` | Theme setup, asset enqueuing, hooks |
| `blocks.php` | Block registration and configuration |
| `helpers.php` | General helper functions |
| `template-tags.php` | Template output functions (pure functions only) |
| `utility.php` | Low-level utilities (asset info, etc.) |

### Hook Registration Pattern

All hook callbacks use a namespace helper:

```php
$n = static function ( $callback_name ) {
    return __NAMESPACE__ . "\\$callback_name";
};

add_action( 'init', $n( 'init' ) );
add_filter( 'the_content', $n( 'filter_content' ) );
```

## Asset Loading

Assets use WordPress's manifest-based loading:

```php
Utility\get_asset_info( 'frontend', 'dependencies' );
Utility\get_asset_info( 'frontend', 'version' );
```

This reads `dist/js/frontend.asset.php` generated by 10up Toolkit during build.

## File Conventions

| Type | Location | Pattern |
|------|----------|---------|
| Blocks | `includes/blocks/{name}/` | block.json, markup.php, edit.js, style.scss |
| Components (CSS) | `assets/css/components/` | `{name}.scss` → import in frontend.scss |
| JS Features | `assets/js/frontend/features/` | `{name}.js` → import in frontend.js |
| Page Templates | `templates/` | `page-{name}.php` |
| Classes | `includes/classes/` | PSR-4 namespace |
| Block Patterns | `patterns/` | `{name}.php` |

## Build System

Uses 10up Toolkit (Webpack-based):

- Entry points defined in `package.json` under `10up-toolkit.entry`
- Block assets auto-discovered from `includes/blocks/*/block.json`
- Output to `dist/` directory
- Supports HMR via `npm run start` (port 5000)
